I've been investigating V4L2 stateless HW video decode on Rockchip RK3588 (Rock 5B+, Mali-G610 MC4, Panfrost driver, Mesa 26.0.4, kernel 7.0-rc3+, Chromium 148.0.7774.0 cross-compiled with use_v4l2_codec=true).

V4L2VideoDecoder is correctly selected and all codecs are listed in chrome://gpu (H.264, HEVC, VP9, VP8 via rkvdec2/hantro, AV1 via hantro). However, decoded frames never reach the compositor. Chromium falls back to software decode for all videos.

After testing three different approaches, I identified two driver-level blockers in the PickDecoderOutputFormat() code path:

PATH A (V4L2 MMAP + EXPBUF):
The rkvdec2 driver does not support VIDIOC_EXPBUF. MMAP buffers cannot be exported as DMABUF fds, so NativePixmapFrameResource creation fails silently. Verified with:

  REQBUFS MMAP: caps=0x0000001d (MMAP=yes, DMABUF=yes, REQUESTS=yes)
  VIDIOC_EXPBUF: errno 25 (ENOTTY) — not supported

PATH B (V4L2 DMABUF import via DmabufVideoFramePool):
Panfrost (Mesa 26.0.4) cannot allocate NV12 buffers via GBM. gbm_bo_create fails for all YUV formats (NV12, YU12) with any usage flags. Only RGBA formats succeed. This means DmabufVideoFramePool cannot provide DMABUF buffers for V4L2_MEMORY_DMABUF on the CAPTURE queue.

The #elif BUILDFLAG(IS_LINUX) && BUILDFLAG(USE_V4L2_CODEC) section in PickDecoderOutputFormat() assumes one of these paths works. On RK3588 + Panfrost, neither does.

Suggested improvement: the Linux V4L2 code path could check for EXPBUF support (via VIDIOC_REQBUFS capabilities) before choosing the MMAP path, and verify GBM format support before choosing the DMABUF path. When neither works, falling through to the ImageProcessor for NV12->ARGB conversion via libyuv would be the correct fallback.

I also found two build issues when compiling Chromium 148 with use_v4l2_codec=true:

1. media/gpu/v4l2/mt21/mt21_decompressor.h:73 — chromium-style error: complex destructor with inline body. Fix: move ~MT21DecompressionJob() = default to the .cc file.

2. media/gpu/v4l2/mt21/mt21_decompressor.cc — multiple -Wunsafe-buffer-usage errors promoted to errors by -Werror. Fix: add #pragma allow_unsafe_buffers or add the path to unsafe_buffers_paths.txt.

Pre-built ARM64 binary for testing: https://github.com/dongioia/rock5bplus-rkvdec2/releases/tag/chromium-v4l2-148

The build is forward-compatible — when Mesa/Panfrost adds NV12 GBM support or the kernel rkvdec2 adds EXPBUF, HW decode will activate without rebuilding Chromium.
