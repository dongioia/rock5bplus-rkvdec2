Hey everyone! I've been thinking about something that could really benefit RK3588 users.

You know how CachyOS builds separate repos optimized for Zen3/Zen4 with architecture-specific compiler flags? We could do the same for RK3588 boards.

The idea: an overlay repo with key packages compiled using -march=armv8.2-a+crypto+fp16+dotprod instead of generic armv8-a. Stock ALARM packages leave a lot of performance on the table because they target the lowest common denominator.

How much difference does it make? I benchmarked this on my Rock 5B+ with just the kernel KCFLAGS change:
  AES-256:  +17.3% (hardware crypto instructions)
  SHA256:   +18.8% (hardware SHA + better atomics)
  memcpy:   +6.1%  (LSE atomics, better codegen)
  No regressions on any metric.

These gains come from the compiler actually using what the hardware has: AES/SHA crypto extensions, LSE atomics (faster locks), FP16, and dot-product instructions. Every RK3588 board has these — they're part of the ARMv8.2-A spec.

The approach would be a small overlay repo (beryllium-rk3588 or similar) with just the performance-critical packages:
  - openssl (crypto everywhere)
  - ffmpeg (media decode/encode)
  - mesa (GPU driver)
  - zstd + lz4 (ZRAM, pacman, btrfs compression)
  - the kernel (already done)
  - chromium (building one with V4L2 right now actually)

Maybe 10-15 packages total. Everything else falls through to the standard Beryllium/ALARM repos — the ABI doesn't change with -march, so optimized and generic packages are fully compatible.

For the build infra, GitHub Actions now offers free ARM64 runners for public repos (4 vCPU). Not fast for Chromium, but fine for smaller packages. We could set up a weekly CI that rebuilds when upstream updates.

The cool part: a detection script at install time checks for RK3588 and enables the repo automatically — same pattern CachyOS uses.

What do you think? Worth pursuing as a Beryllium feature for RK3588 boards, or too much maintenance overhead for the team?
